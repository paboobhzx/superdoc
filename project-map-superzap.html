SuperZAP is a multi-tenant SaaS CRM that integrates WhatsApp for conversation management at scale. Each company (tenant) connects their WhatsApp number, manages operators, tracks leads through a sales pipeline, and monitors real-time customer service metrics — all tenant-isolated with zero data leakage between accounts.

Role: Sole architect and engineer — architecture, backend, frontend, infrastructure, DevOps, testing.
Timeline: 20+ development rounds, 40+ commits, from scratch to production.
Status: In production serving real businesses.

! Challenges Encountered

Real engineering problems encountered during development and how they were resolved.

🔴 WhatsApp API Upgrade — Docker Build with Insufficient RAM
Problem: Old version blocked. No pre-built Docker image available. On EC2 (4GB RAM), build consumed 100% CPU/IO, killed SSM agent, OOM killer terminated process. SSM timeout (600s) shorter than build time (15-20 min).
Solution: 2GB swap (total 6GB), background build with nohup, Docker cache cleanup, container inspection for correct credentials.
⏱ ~3 hours lost
🔴 Webhook URL: localhost vs Docker Gateway
Problem: Webhook configured with localhost:3000. Inside Docker, localhost is the container itself, not the EC2 host. All events: ECONNREFUSED. Bug replicated in 5 files.
Solution: Use Docker network gateway. Rule: NEVER use localhost in URLs that Docker needs to reach the host. Global grep and fix in 5 files.
⏱ Recurred in SZR04 + SZR05
🟡 Auth Interceptor Captured Login Fallback 401
Problem: Axios interceptor captured ALL 401s, including the intentional one during auth fallback. Triggered redirect to /login before second login completed. Infinite loop.
Solution: Exclude auth routes from interceptor. Rule: auth routes exempt from automatic 401 handling.
🟡 Login Didn't Return Tenant ID — Chat in Infinite Loading
Problem: Endpoint returned token and name but not ID. Frontend needed ID to load conversations. Appeared intermittent (worked after refresh).
Solution: Add missing field to response. Simple bug, hard to diagnose.
🟡 Conversations Not Loading — Query on Non-Existent Field
Problem: Controller queried a non-existent instance field in the multi-operator architecture. Empty list, empty stats.
Solution: Alternative repository functions querying by tenant_id with legacy compatibility fallback.
🔵 PM2 Didn't Persist Between Reboots
Problem: Node.js process didn't start after reboot. PM2 startup conflicted with environment variables.
Solution: Standardize PM2_HOME across all deploy commands, pm2 save after each deploy.
🔵 Nginx + SSL + CORS — Mixed Content
Problem: HTTPS frontend calling HTTP backend. Browser blocked due to mixed content and CORS.
Solution: Nginx reverse proxy + auto-renewable Let's Encrypt. Long WebSocket timeout for Socket.IO.
🔵 SSH Key — EC2 with Different Key Pair
Problem: EC2 created via Terraform with specific key pair. New keys had no access.
Solution: Use SSM exclusively. Eliminate SSH dependency.

7 Architecture Decisions

EC2 + Express vs Serverless
Choice: EC2 with Node.js/Express.
Reason: WebSocket (Socket.IO) requires persistent connections. WhatsApp Docker API needs to run on the same host.
MySQL vs DynamoDB
Choice: MySQL (RDS).
Reason: Multi-tenancy with foreign keys and JOINs is more natural in relational. Per-tenant isolation with complex queries.
WhatsApp API in Docker
Choice: Container on the same EC2.
Reason: Minimal latency for webhooks, full control, no dependency on paid external services.
Modular Architecture (11 domains)
Choice: Controller/Service/Repository per domain.
Reason: Independence, testability, tenant isolation at the repository level.
3-Level Auth
Choice: Super Admin + Tenant Admin + Operator.
Reason: SaaS needs a global admin. Companies need a manager. Operators with limited access and temporary tokens.
Terraform IaC (8 modules)
Choice: Infrastructure as Code.
Reason: Reproducibility, versioning. All infra recreatable from scratch with a single command.
